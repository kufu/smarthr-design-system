# smarthr-ui アップグレードとドキュメント追従

smarthr-uiを最新バージョンにアップグレードし、リリースノートに基づいてデザインシステムサイトのドキュメントを修正する。

## 手順

### ステップ1: 現在のバージョンを記録

`package.json` を読み取り、現在の `smarthr-ui` のバージョンを確認・記録する。

### ステップ2: アップグレード実行

```bash
pnpm upgrade:smarthr-ui
```

完了後、`package.json` から新バージョンを読み取る。バージョンが変わっていない場合は「すでに最新バージョンです」と報告して終了する。

### ステップ3: 差分のリリースタグ一覧を取得

旧バージョンから新バージョンまでの間にリリースされたタグを特定する。

```bash
gh release list --repo kufu/smarthr-ui --limit 50 --json tagName,publishedAt
```

タグ名の形式は `smarthr-ui-vX.Y.Z`。旧バージョン（含まない）〜新バージョン（含む）の範囲のタグを古い順に並べる。

### ステップ4: 各リリースノートを取得

対象タグそれぞれについてリリースノートを取得する。

```bash
gh release view {タグ名} --repo kufu/smarthr-ui --json body,tagName,publishedAt
```

### ステップ5: 対応事項を分析

取得したリリースノートから以下を**すべて**確認し、ドキュメントへの影響を整理する：

1. **「プロダクト側で対応が必要な事項」セクション**を最優先で確認する
2. **「⚠ BREAKING CHANGES」セクション**を確認する。このセクションは「プロダクト側で対応が必要な事項」に記載されていない場合があるため、必ず独立して確認すること
3. **「Bug Fixes」セクション**でドキュメントのコード例に影響するものがないか確認する（ワークアラウンドの削除指示などが書かれている場合がある）
4. **「Features」セクション**で新コンポーネントや新しいバリアントが追加されていないか確認する

ドキュメントへの影響として考えられるもの：
- 既存コンポーネントのprops・挙動の変更 → 対応するindex.mdxのコード例・説明を修正
- propsの削除（BREAKING CHANGES）→ 対応するindex.mdxのコード例から削除済みpropsを除去し、代替手段を説明する
- 新コンポーネントの追加 → 新しいドキュメントページを作成
- コンポーネントの削除・非推奨化 → ドキュメントに記載を追加
- paddingやスタイルのデフォルト値の変更 → コード例の調整が必要な場合がある
- ワークアラウンドの削除指示（Bug Fixesに記載されることがある）→ ドキュメントのコード例に該当コードがあれば削除する

### ステップ6: ドキュメントを修正する

分析結果をユーザーに提示し、対応内容を確認してから修正を進める。

**既存ページの修正：**
- 対象ファイル: `src/content/articles/products/components/{name}/index.mdx`
- リリースノートの指示に従い、コード例・説明文を修正する

**新コンポーネントのページ作成：**
- `src/content/articles/products/components/{ケバブケース名}/` ディレクトリを作成
- 既存ページ（例：`error-screen/index.mdx`）を参考にして `index.mdx` を作成する
- Chromaticのリンクがリリースノートにある場合はコンポーネントの見た目・propsの参考にする
- `smarthr-ui` の`node_modules`にある型定義も参考にしてpropsを把握する

### ステップ7: 作業サマリーを報告

完了後、以下を報告する：
- アップグレードしたバージョン（旧→新）
- 対応したリリースの一覧
- 修正・作成したファイルの一覧と変更内容の概要
- 対応が必要だが自動化できなかった事項（あれば）
